# llm.txt — Hermes + Pinto project guide for AI agents

This repo runs Hermes Agent as a Pinto chatbot.

Primary goal:
- Start Hermes Gateway.
- Expose Pinto webhook at `/plugins/pinto/webhook`.
- Configure AI provider so Hermes can answer messages.
- Let users chat with Hermes through Pinto.

Supported runtimes:
- Podman on macOS/Linux.
- Docker on macOS/Linux.
- Docker Desktop on Windows via PowerShell.

Important files:
- `README.md`: user-facing setup guide from zero to running.
- `.env.example`: environment template.
- `.env`: local secrets/config, do not commit.
- `Dockerfile`: Docker image.
- `Dockerfile.podman`: Podman image.
- `docker-compose.docker.yml`: Docker compose with gateway + dashboard + cli.
- `docker-compose.podman.yml`: Podman compose with gateway + dashboard + cli.
- `docker-compose.trycloudflare.yml`: Cloudflare Quick Tunnel overlay.
- `docker-compose.cloudflare.yml`: Cloudflare named tunnel overlay.
- `scripts/bootstrap-hermes-config.py`: writes Hermes config under `hermes-config/`.
- `scripts/patch-api-server-for-pinto-podman.py`: Hermes API server compatibility patch.
- `scripts/patch-pinto-for-podman.py`: Pinto adapter compatibility patch.

Key URLs:
- Local Gateway health: `http://127.0.0.1:8642/health`
- Local Pinto webhook: `http://127.0.0.1:8642/plugins/pinto/webhook`
- Hermes Dashboard: `http://127.0.0.1:9119/`
- Public Pinto webhook: `https://<cloudflare-host>/plugins/pinto/webhook`

Required first-start `.env` values:
```env
PINTO_BOT_ID=your_pinto_bot_id
API_SERVER_KEY=generated_hex_key
API_SERVER_HOST=0.0.0.0
PINTO_ALLOW_ALL_USERS=true
GATEWAY_ALLOW_ALL_USERS=true
HERMES_DASHBOARD=1
HERMES_DASHBOARD_PORT=9119
```

Persistent Pinto bot ID behavior:
- `scripts/bootstrap-hermes-config.py` treats `hermes-config/config.yaml` `platforms.pinto.extra.botId` as the durable source of truth.
- `PINTO_BOT_ID` from process env or `/root/.hermes/.env` is only a fallback when `config.yaml` has no botId yet.
- Extra Pinto bots are configured under `platforms.pinto.extra.bots` and can point to persona keys under `platforms.pinto.extra.pintoAgents`.
- Persona routing is local Hermes-Pinto behavior: `bots.<bot_id>.persona` resolves to `pintoAgents.<key>.channelPrompt`. This is not Hermes native subagent registry.
- All Pinto bots share one webhook URL and webhook secret; only `bot_id` and optional `persona` differ.
- Do not commit real bot IDs or secrets; `hermes-config/` and `.env` are ignored.

Local Hermes-only multi-agent handoff (no OpenClaw involved):
- `scripts/hermes-company-run.py <task> [--agents key1 key2 ...]` calls Hermes
  `/v1/responses` once per persona key, sequentially, feeding each reply as
  the next persona's input (real handoff chain, not just persona switching).
- Persona keys and prompts come from `platforms.pinto.extra.pintoAgents`.
- Default chain order can be set at `platforms.pinto.extra.companyWorkflows.default`.
- Needs `API_SERVER_KEY` from `.env`/`hermes-config/.env` and the gateway
  reachable at `http://127.0.0.1:8642` (override with `--api-base`).
- Run history is written to `hermes-config/company-runs/` (ignored by git).
- Pinto bots can be connected directly by adding `companyWorkflow: [persona1, persona2, ...]` to `platforms.pinto.extra.bots.<bot_id>`; the adapter intercepts that bot's webhook, runs the chain, and sends the final output back to the Pinto chat.
- This is a custom layer on top of Hermes; Hermes core has no native
  persistent agent registry or agent-to-agent tool like OpenClaw.

Notes:
- Requires Hermes OAuth login via `hermes setup --portal` in the Hermes environment.
- Verify with `hermes portal info`.
- The exact `:free` suffix is required for the free-tier variant.
- The offer was announced as limited-time; if unavailable, check Nous Portal model picker.

Do not print or commit real API keys.

Cloudflare Quick Tunnel behavior:
- Quick Tunnel URLs are temporary.
- Restarting cloudflared or running compose down may change URL.
- When URL changes, update Pinto Developer Console webhook URL.
- For stable production URL, use Cloudflare named tunnel and `CLOUDFLARE_TUNNEL_TOKEN`.

Common expected health responses:
```json
{"status":"ok","platform":"hermes-agent"}
```

```json
{"ok":true,"channel":"pinto"}
```

If Pinto webhook returns 500:
- Check gateway logs first.
- Verify `platforms.pinto.extra.botId` in `hermes-config/config.yaml`.
- Verify `PINTO_BOT_ID` only as first-start fallback.
- Verify API server host is `0.0.0.0`.
- Verify Pinto adapter compatibility patches are included in image.
- Verify AI provider config if logs say `No inference provider configured`.

If AI provider test is needed:
```bash
hermes -z "reply only: ok" --provider openai-api --model codex
```
Expected output:
```text
ok
```

If logs show:
```text
Pinto HTTP 500: {"ok":false,"error":"Failed to send message"}
```
Hermes received webhook and likely generated reply, but Pinto rejected send-back request. Check real Pinto `chat_id`, bot ID, Pinto API URL, webhook secret, and bot permissions.

Important implementation notes:
- Hermes Gateway must bind to `0.0.0.0` inside container.
- Dashboard is separate from Gateway and runs on port `9119`.
- Gateway root `/` may return 404; use `/health` for health checks.
- Pinto webhook route is `/plugins/pinto/webhook`.
- `host.docker.internal:host-gateway` can break Podman on macOS; Podman compose avoids it.
- The compatibility patches fix Hermes 0.16 route mounting, `response_class`, and Pinto adapter `send()` signature.

When editing:
- Keep Podman and Docker compose files separate.
- Keep user-facing instructions simple in `README.md`.
- Do not commit `.env` or secrets.
- Prefer adding helper scripts over telling new users long compose commands.
